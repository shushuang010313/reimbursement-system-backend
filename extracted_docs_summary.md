## 2.1 需求概述

### 2.1.1 业务背景

员工因公出差，或为企业办理业务，自己先垫钱消费，业务结算后与企业进行报账结算；也有员工提前向公司借款，办理业务完成后进行多退少补的结算。如果员工出差全部走线上预订，由公司/服务商/供应商垫付资金，员工出差回来后拿相关票据报账，此时报销金额为0，企业财务对服务商/供应商进行对账结算。

### 2.1.2 需要解决的财务痛点

审批耗时耗力：员工携带贴票报销单逐级审批，财务人员审核时需考虑多维因素，人力与时间成本高

业务真实性难以保障：消费时间真实性（行程是否重复）、消费金额真实性（是否与发票金额一致）、费用真实性（是否有无关发票）

票据合规性难以验证：查真查重、发票抬头与税号一致性、连号检测、发票有效期检查

财务与员工矛盾：人工审批时间慢，票据问题多，打回重提单据频繁，员工体验差

### 2.1.3 需求概述

报销单列表功能：支持多条件查询（报销单号、标题、事由、费用归属公司、报销部门、报销人、业务类型）及列表展示

报销单据页面：包含单据头部、基本信息、补录行程、补助信息、补助日历、费用合计、费用分摊、备注信息等功能分区

补助自动计算：根据出差城市等级自动匹配餐费/交通/通讯补助标准，支持日历维度勾选管理

费用分摊管理：支持多行费用分摊，第一行比例自动联动计算，确保总比例100%

单据状态管理：草稿（0）→ 已完成（1）→ 已作废（2）

5. 关键技术点

## 5.1 并发编程

补助日历数据生成：当补录行程条数较多时，使用CompletableFuture异步并行生成各行程的补助日历数据，提升响应速度

分布式锁：报销单提交时，使用Redis分布式锁（redisson）防止重复提交

线程安全：所有共享状态使用线程安全对象（AtomicLong、ConcurrentHashMap等）

业务防抖拦截：在报销单提交接口挂载自定义注解 @Idempotent，利用 AOP 切面拦截请求，以 用户 Token + 请求体参数 MD5 为 Key 动态生成请求指纹，利用 Redis + Lua 脚本 执行原子性判定，在 5 秒防抖期内拦截相同的重复请求，从入口端杜绝重复单据的产生。

资源并发锁控制：若请求通过防抖拦截，在实际执行状态机流转时，使用 Redis 分布式锁（Redisson）锁定当前报销单 ID（reimId），防止多节点集群环境下对同一张报销单的并发修改冲突。

## 5.2 事务控制

### 5.2.1 数据库局部事务

以下场景需加 @Transactional 注解保证原子性：

保存报销单：主表 + 补录行程 + 补助信息 + 补助日历 + 费用分摊 需在同一事务中

删除补录行程：级联删除补助信息和补助日历需在同一事务中

提交报销单：状态变更 + 日志记录需在同一事务中

### 5.2.2 分布式事务

报销单提交后若需推送BPM审批流，采用最终一致性方案：

本地事务保存报销单数据，同时写入待推送MQ消息表

通过补偿机制（定时Job扫描）保证MQ消息最终成功发送

BPM回调失败时，系统自动重试，记录重试日志

发送端确认机制（Publisher Confirm）：发送端采用本地事务出件箱模式，将报销数据保存与消息发送记录绑定在同一数据库事务中。同时，开启 RabbitMQ 的发送确认模式与退回监听机制。只有当消息确实到达 RabbitMQ 代理且成功持久化后，系统才更新发送记录为成功，确保消息“绝不丢失”。

消费端幂等接收：BPM 回调监听端通过全局单据号和版本号执行防重校验，防止消息重复消费导致审批流被重复触发。

## 5.3 Job使用

本项目主要使用 Spring Boot 自带的 `@Scheduled` 注解实现轻量级定时任务。目前系统中的核心 Job 为 **BPM审批流推送补偿任务**（`BpmCompensationJob`）。

### 1. 任务触发与调度
- **执行频率**：每 5 分钟执行一次（Cron 表达式：`0 */5 * * * ?`）。
- **调度方式**：基于 `@Scheduled` 注解和 `@EnableScheduling` 开启的定时任务调度。

### 2. 核心业务逻辑
- **数据扫描**：扫描报销单主表（`fk_reim_main`）中，状态为 `1`（已完成/待推送）且最后更新时间超过 5 分钟的单据，以寻找可能推送失败的滞留数据。
- **防止重复推送**：通过 Redis 检查 Key（`fk:reim:pushed:{reimId}`），如果该标志存在，说明已经推送成功过，直接跳过。
- **重试机制与阈值控制**：通过 Redis 记录单据的重试次数（`fk:reim:retry:{reimId}`）。如果某张单据的重试次数达到 3 次及以上，系统将不再自动重试，而是记录 Error 级别日志触发报警，等待人工介入处理。
- **消息补发**：对满足补偿条件的单据，重新向 RabbitMQ（`reim.submit.topic`）投递报销单 ID。
- **状态记录**：补偿推送成功后，在 Redis 中写入成功标志并设置 30 天的过期时间；若补偿推送抛出异常，则将该单据在 Redis 中的重试次数加一。

此补偿任务作为 MQ 可靠消息投递和重试机制的重要一环，保证了在瞬时网络抖动或服务重启等异常情况下，意外未能成功推送到 MQ 的报销单依然能够被及时发现并重推，确保了审批流程的最终一致性和系统的可靠性。

## 5.6 敏感信息处理

数据库透明加密设计：在持久层（ORM）通过字段类型转换机制，在敏感数据写入数据库时自动进行对称加密，在读取时自动解密。即使数据库文件被恶意复制或泄露，攻击者也只能看到密文。

前端传输动态脱敏设计：通过在实体类敏感字段上声明自定义的脱敏注解，并配合定制化的 JSON 序列化处理器，使得接口在向前端返回数据时自动解析注解。序列化器会根据预置的敏感字段分类规则（如姓名、身份证、电话、卡号等），在内存中动态将中段字符替换为掩码字符（如星号），确保到达浏览器控制台和前端展示的数据均为非明文。

日志非侵入打码：配合切面技术，在系统的控制台日志与异动日志中自动对带有敏感标记的参数进行识别和数据脱敏，防止开发或运维人员通过日志审计系统间接获取员工隐私。

## 5.7 全局异常与错误码规范

5.7.1全局异常拦截架构

本系统废弃了传统的在 Controller 层到处写 try-catch 的臃肿做法，采用 Spring Boot 的 @RestControllerAdvice 结合 @ExceptionHandler 实现了全局统一异常拦截。

参数校验异常：自动拦截基于@Validated 触发的MethodArgumentNotValidException，转化为 400 参数错误并返回详细校验信息。

业务逻辑异常：在 Service 层遇到逻辑阻断时，直接 throw new BusinessException。全局切面会捕获该异常，并自动包装为标准的统一 JSON 返回体（Result<T>）输出给前端。

系统未知异常：统一兜底拦截，转化为 500 系统内部异常，防止后台代码运行报错导致堆栈信息外泄（防渗透）。

## 5.9 大数据量问题

低内存开销引擎（EasyExcel）：采用阿里开源的 EasyExcel，利用其底层的 SAX 流式写入原理，不保留全量数据对象在内存中，而是采用边读边写、按行解析并将临时缓存写入磁盘的低内存消耗方案，使百万级导出所需的固定内存稳定在十几兆字节。

异步线程池处理（Decoupled Async Work）：导出任务由前端触发后，主线程不等待结果，而是直接在后台自定义的高效线程池中排队处理。

分批次分页抓取：异步任务内部使用分页查询或游标读取方式，从数据库中分批次获取数据写入 Excel。写入完成后，将生成的 Excel 文件直接同步至公司对象存储服务器（OSS）中。

异步生命周期轮询：前端提交导出后即刻返回一个全局任务 ID。后端在 Redis 中记录该任务的状态（处理中、成功、失败）。前端通过定时轮询任务状态接口（或通过 WebSocket）获取导出进度。当任务在 Redis 中的状态更新为“成功”时，前端直接跳转到返回的文件下载 URL，彻底保障了系统的稳定性和响应体验。

## 5.10 缓存的使用

计算引擎基础数据缓存：在报销单核心的补助计算引擎中，城市等级是影响计算结果的高频读取且极少变更的基础数据。系统采用了基于 Redis 的统一缓存策略，并为其设置了合理的 24 小时过期时间。这一机制有效避免了报销单批量流转、计算补助时反复拉取静态数据的开销，大幅提升了引擎的并发计算性能。

Redisson 分布式锁防重复提交：针对报销单提交（REIM_Submit）等资金敏感型接口，考虑到弱网环境下的前端频繁连点问题，除了前端的防抖控制，系统底层引入了 Redisson 分布式锁机制。以 fk:reim:lock:submit:{reimId} 为锁的 Key，实现了服务接口级的强幂等性，从根本上杜绝了脏数据的产生。

## 5.11 MQ的使用

核心链路异步解耦：当报销单完成所有的本地校验与状态变更后，由于启动下游的 BPM 审批流属于较重的操作，系统引入了 RabbitMQ 进行解耦。报销单提交后，系统立即返回成功，并在后台向 reim.submit.topic 投递异步消息触发审批流。这样既避免了因为同步调用 BPM 导致的接口长时间阻塞，又极大地提升了系统的吞吐量与用户体验。

闭环事件驱动架构：基于业务生命周期闭环的考量，审批流的完结、打回以及资金结算等后置操作，同样依托 MQ 事件流驱动。它与我们在 5.12 中设计的死信队列（DLX）防丢机制相辅相成，打造了具有企业级可靠性的消息通信骨架。

死信队列（DLX & DLQ）沉淀：如果重试 3 次后该报销消息依然推送失败，则将其移出主队列，通过死信交换机路由转发到专用的死信队列（Dead Letter Queue）中进行存储，防止堵塞主消息队列的正常流转。

人工干预与自动报警机制：配置专用的死信消息监听器。一旦死信队列中积压了投递失败的单据，监听器会自动触发钉钉机器人或邮件接口报警，将单据 ID、重试次数和错误堆栈推送到财务系统开发运维群。财务人员可以通过系统提供的管理后台进行一键重推和状态修复，保障账目 100% 不丢单。

## 5.12 重试机制

BPM回调失败时，系统自动重试，记录重试日志

死信队列防丢机制： 若推送到 BPM 的消息超过 3 次重试仍未成功，消息将被拒绝并自动路由到死信交换机。 后台监听死信队列，一旦有消息落入，立即记录，并调用通知服务向运维群发送告警，确保异常单据第一时间得到人工干预，达成业务闭环的 100% 最终一致性。

故障自动重试：当 BPM 审批系统因瞬时网络抖动或服务部署而不可用，导致消息投递或消费失败时，系统将触发 3 次指数退避重试（即逐步拉长重试间隔，如 1秒、3秒、10秒），避开临时网络瓶颈。

## 5.13 补助计算引擎弹性设计

痛点与动机：传统的硬编码分支条件在面对频繁变化的差旅补助标准（如根据特殊城市等级调整、新增高管特殊补助、或者特定业务类型临时补贴）时，需要频繁修改计算主类，容易破坏已有逻辑，违背了“开闭原则（OCP）”。

策略模式应用：将标准补助计算逻辑抽象为统一的策略规范接口。针对不同城市等级的餐补、交通补、通讯补计算逻辑（如一线城市计算规则、二线城市计算规则、三线城市计算规则等）分别实现独立的计算策略类。各策略类之间相互隔离，各司其职。

简单工厂与 Spring 容器装配：构建一个补助策略工厂类。应用启动时，利用容器的依赖注入特性将所有的计算策略自动注册进工厂的策略字典中。当核心计算引擎 SubsidyCalcEngine 执行计算时，通过解析当前行程的目的地城市等级，直接从工厂中动态检索并调用对应的具体策略。未来如需新增海外补贴、重大活动临时补贴等规则，开发人员仅需编写新的策略类，对已有系统零侵入。

操作 | 数据流向

全链路鉴权 | 前端 → Auth 拦截器 → 绑定至全局 UserContext 线程上下文 → 报销服务

新增报销单 | 前端 → Auth 拦截器 →报销服务 → MySQL(fk_reim_main INSERT)

保存补录行程 | 前端→Auth 拦截器 → 报销服务 → MySQL(fk_reim_trip INSERT/UPDATE + fk_reim_subsidy INSERT + fk_reim_calendar INSERT) → 事务提交

修改补助日历 | 前端→ Auth 拦截器 →报销服务 → MySQL(fk_reim_calendar UPDATE) → 联动更新fk_reim_subsidy金额 → 联动更新fk_reim_main合计金额

提交报销单 | 前端→ Auth 拦截器 →报销服务 → Redis(分布式锁) → 数据校验 → MySQL(fk_reim_main UPDATE status) → 写MQ消息表 → 释放锁 → MQ异步推送BPM

查询列表 | 前端 → Auth 拦截器 → 报销服务 → DataPermissionInterceptor（MyBatis 拦截器提取 UserContext，自动拼接 WHERE reimburser_id = ? 隔离数据） → MySQL(SELECT 分页) → 组装 VO → 返回

审批回调 | BPM 审批流系统 → 验签拦截器（校验防篡改签名） → 报销服务 → 更新主单状态 → AOP 异动日志切面（记录审批记录） → MySQL
