# 答辩评委提问预判 Q&A

> 组长：舒爽 | 基于已实现功能的全量问答覆盖

---

## 一、状态机与主流程

### Q1：报销单的状态机是怎样设计的？有哪些状态？流转规则是什么？

**A：** 三种状态：0-草稿、1-已完成（待推送BPM）、2-已作废。

流转规则：
- 新建 → 0（草稿），可反复编辑保存
- 0 → 提交校验通过 → 1（已完成），同时触发推送到 BPM 审批流
- 1 → 作废 → 2，已完成的单据才能作废
- 草稿不能直接作废，必须先提交再作废，这样设计是为了防止误删有效数据

另外，BPM 审批回调 `updateStatus` 可以驱动状态的后续流转（比如审批通过后变为完成态、驳回后回退等），这部分由 BPM 系统通过签名验证接口来驱动。

---

### Q2：为什么草稿不能直接作废？这样设计会不会不方便？

**A：** 这是刻意设计的约束。草稿本身就是临时数据，如果用户不想要了，直接删除即可（我们有逻辑删除机制）。而"作废"是一个审计动作——它意味着"这笔单据曾经有效，现在被撤销了"，需要记录在异动日志里。草稿从未生效过，不需要"作废"这个语义，直接删掉更合理。这符合财务系统的审计追溯要求。

---

### Q3：提交时做了哪些校验？如果校验不通过会怎样？

**A：** 提交时做了四层校验：

1. **状态校验**：必须是草稿状态（0），否则抛出 `REIM_002`
2. **乐观锁版本校验**：前端传入的 version 必须与数据库一致，防止并发覆盖
3. **完整性校验**：必须有行程明细（`tripCount > 0`）且补助总额不为空，否则抛出 `REIM_005`
4. **数据库层面乐观锁**：MyBatis-Plus 自动在 UPDATE 语句追加 `WHERE version = ?`，更新失败则抛并发异常

任何一层校验不通过都会终止提交，返回对应的错误码和提示信息。

---

### Q4：updateStatus 为什么不做状态机校验？会不会出现非法的状态流转？

**A：** `updateStatus` 是给 BPM 审批流回调用的，它的设计理念是"信任上游"——BPM 系统通过 `SignAuthInterceptor` 签名验证后才能调用，是一个内部可信通道。如果在这里加状态机校验，会导致 BPM 和报销系统的状态定义强耦合，任何一方的状态变更都会影响对方。

实际项目中，状态合法性应该由 BPM 流程定义来保证，报销系统作为下游，职责是忠实地记录状态变更并记入审计日志。当然，如果要做防御性编程，加一层白名单校验会更安全，这是我们后续可以优化的点。

---

## 二、费用分摊倒挤法

### Q5：倒挤法的核心思路是什么？为什么不用普通的比例乘法？

**A：** 核心思路：**指定第一行（sortNo=1）为兜底行，其他行正常算，兜底行用减法反推。**

具体来说：
- 其他行：`金额 = 总金额 × 比例`，正常四舍五入
- 兜底行：`金额 = 总金额 - 其他行金额之和`，纯减法

为什么不用普通乘法？因为浮点精度问题。比如 100 元分给 3 个人，每人 1/3：
- 普通乘法：33.33 × 3 = 99.99，亏了 0.01
- 倒挤法：行2=33.33, 行3=33.33, 行1=100-66.66=33.34，总和=100.00

在财务系统里差一分钱就是账不平，倒挤法从数学上杜绝了这个问题。

---

### Q6：如果其他行的比例之和已经超过 100% 怎么办？

**A：** 代码里有校验：`if (totalOtherRatio > 1.0)` 直接抛出 `REIM_004`（分摊比例超过 100%）。这个校验在计算兜底行之前执行，防止出现负数兜底。

---

### Q7：兜底行的金额有可能是负数吗？

**A：** 理论上，如果比例校验被绕过，兜底金额可能为负。但我设了两层保护：
1. 比例上限校验（totalOtherRatio > 1.0 直接报错）
2. 前端将第一行设为只读，用户无法手动修改

目前没有对兜底金额做负数兜底校验，如果评委追问，可以坦诚说这是后续可以加固的点——加一行 `if (firstAmount < 0) throw Exception` 即可。

---

### Q8：分摊比例用 BigDecimal 还是 double？精度怎么控制的？

**A：** 全部使用 `BigDecimal`。比例用 `setScale(4, HALF_UP)` 保留 4 位小数，金额用 `setScale(2, HALF_UP)` 保留 2 位小数。Java 的 `BigDecimal` 是精确十进制运算，不会产生二进制浮点误差，这是财务系统的基本要求。

---

### Q9：如果用户只填了一行分摊，倒挤法还生效吗？

**A：** 如果只有一行且 sortNo=1，那么 `totalOtherRatio = 0`，兜底行的比例 = 1.0，金额 = 总金额，整笔费用全部分摊到这一行，逻辑正确。如果只有一行但不是 sortNo=1（比如 sortNo=2），兜底行不存在，其他行正常计算，金额等于总金额，也不会出错。

---

## 三、AOP 异动日志

### Q10：异动日志是怎么实现的？为什么不用直接在方法里写日志？

**A：** 使用 Spring AOP 的 `@Around` 环绕通知，拦截 `submitReim`、`cancelReim`、`updateStatus` 三个方法。

流程是：
1. 方法执行前：从数据库查一次，保存旧快照
2. 执行目标方法
3. 方法执行后：再查一次，拿到新快照
4. 比对 `oldStatus != newStatus`，状态真的变了才写日志

不用在方法里直接写日志的原因是：**关注点分离**。核心业务代码不应该掺杂日志逻辑，用切面统一处理更干净、不易遗漏，而且未来要改日志格式或追加日志内容，只改切面一个地方就行。

---

### Q11：日志写入失败会影响业务吗？

**A：** 不会。日志写入被包裹在 try-catch 里，失败了只打 error 日志，不抛异常。这是刻意设计的——审计日志是辅助功能，不能因为磁盘满或数据库故障导致报销单提交失败。业务正确性优先于审计完整性。

---

### Q12：日志里记录了哪些字段？有没有考虑敏感信息脱敏？

**A：** 记录字段：`reimId`、`action`（提交/作废/状态更新）、`oldStatus`、`newStatus`、`details`（JSON 含单号、报销人姓名、金额）、`operatorId`、`operatorName`。

目前日志的 details JSON 是通过 `ObjectMapper.writeValueAsString` 直接序列化的，没有走 `@JsonEncrypt` 脱敏注解。报销人姓名以明文存入日志。如果评委指出这一点，坦诚说这是一个安全待优化项——应该在日志序列化时复用脱敏逻辑。

---

## 四、乐观锁 @Version

### Q13：乐观锁和悲观锁的区别？你为什么选乐观锁？

**A：**

| | 乐观锁 | 悲观锁 |
|---|--------|--------|
| 原理 | 更新时检查版本号，冲突就失败 | 读取时就加锁，其他人等锁释放 |
| 性能 | 无锁等待，高并发表现好 | 有锁等待，并发高时排队 |
| 适用场景 | 写冲突少的场景 | 写冲突多的场景 |

我选乐观锁是因为报销单的编辑场景：同一个报销单通常只有一个人编辑，并发修改的概率很低。用乐观锁既保证了数据安全，又不牺牲性能。如果真的冲突了，直接提示用户"数据已被他人修改，请刷新重试"即可。

---

### Q14：MyBatis-Plus 的 @Version 是怎么工作的？你做了什么配置？

**A：** 分三步：

1. **实体层**：在 `ReimMain` 的 `version` 字段上加 `@Version` 注解
2. **配置层**：在 `MyBatisPlusConfig` 中注册 `OptimisticLockerInnerInterceptor` 到插件链
3. **SQL 层**：调用 `updateById(entity)` 时，MP 自动将 SQL 改写为：
   ```sql
   UPDATE fk_reim_main SET ..., version = version + 1 
   WHERE id = ? AND version = ?
   ```
   如果 version 不匹配，影响行数为 0，MP 返回 false，业务层捕获后抛异常。

---

### Q15：乐观锁在保存和提交时分别怎么用？

**A：**

- **保存（更新草稿）**：前端必须传 version。服务端 `updateById(main)` 后检查返回值，false 就抛"数据已被其他人修改"。
- **提交**：做了双重校验——先用 `equals` 比对前端传入的 version 和数据库 version，再借助 `@Version` 的自动 SQL 改写做数据库层校验。双重保障。

---

### Q16：cancelReim 方法有用乐观锁吗？

**A：** `cancelReim` 调用了 `this.updateById(updateMain)`，如果传入的 `updateMain` 对象没有设置 version 字段（当前代码确实没设置），MP 的乐观锁拦截器就不会追加 `WHERE version = ?` 条件，等于乐观锁没生效。这是一个待修复的问题——作废操作也应该从 DTO 接收 version 字段，保证并发安全。

---

## 五、UserContext 与数据权限

### Q17：UserContext 是怎么实现用户信息透传的？为什么用 ThreadLocal？

**A：** `UserContext` 封装了一个 `ThreadLocal<String>`，在 `AuthInterceptor.preHandle()` 中从请求头提取 userId 后 set 进去，请求结束时在 `afterCompletion()` 中 clear。

用 ThreadLocal 是因为它和请求线程生命周期绑定——一个请求从进来到出去都是同一个线程，ThreadLocal 正好适合存储请求级的上下文信息。相比之下，如果用方法参数传 userId，每个 Service 方法都要多一个参数，侵入性太强。

---

### Q18：如果忘记 clear 会有什么后果？你怎么防止？

**A：** Tomcat 使用线程池，请求结束后线程不销毁，而是回收到池子里。如果没 clear，下一个复用这个线程的请求就会读到上一个用户的 userId，导致"串号"——用户 A 看到了用户 B 的数据。

我的防护措施：在 `AuthInterceptor.afterCompletion()` 中做了 `finally` 式的 clear。`afterCompletion` 是 Spring 拦截器的生命周期回调，无论请求成功还是异常都会执行，确保一定清理。

---

### Q19：数据权限是怎么实现的？如何保证用户只能看自己的报销单？

**A：** 使用 MyBatis-Plus 的 `DataPermissionInterceptor`，在 `MyDataPermissionHandler` 中动态拼装 SQL 条件。拦截器拦截所有查询请求，自动在 WHERE 子句后追加 `AND reimburser_id = '<当前userId>'`。

当前 userId 从 `UserContext.getUserId()` 获取。如果是 BPM 回调等内部调用（UserContext 为空），则不追加条件，允许查看全部数据。

---

### Q20：@Async 异步任务里能拿到 UserContext 吗？

**A：** 不能自动拿到。`@Async` 方法在独立线程池中执行，不会继承父线程的 ThreadLocal。当前代码的做法是：在 Controller 层手动把 userId 传给异步方法，异步方法内部自己管理 UserContext，finally 中自己 clear。更好的做法是用阿里的 `TransmittableThreadLocal`，可以让线程池中的子线程自动继承父线程的上下文，这是我们后续可以优化的方向。

---

## 六、RabbitMQ 可靠消息（亮点三）

### Q21：你用什么方案保证报销单推送到 BPM 不丢消息？

**A：** 本地消息表 + 可靠消息投递 + 补偿定时任务 + 死信队列兜底，四层保障。

1. **本地消息表**：在报销单提交的同一个数据库事务中，向 `fk_mq_message` 插入一条 status=0 的消息记录。事务保证了"状态改了，消息一定记了"。
2. **即时发送**：事务提交后立即通过 RabbitTemplate 发送，成功→status=1，失败→status=2。
3. **补偿任务**：`BpmCompensationJob` 每 5 分钟扫描一次，找出 status=1（已完成）且超过 5 分钟未被标记为已推送的报销单，补发 MQ。
4. **死信队列**：消费者重试 3 次（指数退避 2s→4s→8s）全部失败后，消息进入死信队列，触发钉钉告警，人工介入。

---

### Q22：为什么不在数据库事务里直接发 MQ？而要分两步？

**A：** 这就是经典的"分布式事务"问题。如果在事务里发 MQ，有两种情况：
- 先发 MQ 再提交事务：MQ 发出去了，但事务回滚了，BPM 收到了一条不存在的单据
- 先提交事务再发 MQ：事务提交了，但 MQ 发送失败了，BPM 永远不知道这笔单据

两种都有问题。本地消息表方案是业界的成熟做法：把消息记录和业务数据在同一个事务里落库，事务成功后再异步发送。即使发送失败，补偿任务会重试。这样保证了最终一致性。

---

### Q23：补偿任务怎么避免重复推送？

**A：** 两重防护：
1. **Redis 幂等标记**：`fk:reim:pushed:{reimId}`，补偿任务发送前检查这个 key，存在就跳过，发送成功后立即写入，TTL 30 天
2. **熔断机制**：`fk:reim:retry:{reimId}` 计数器，最多补偿重试 3 次，超过后钉钉告警不再重试，防止死循环

---

### Q24：补偿任务在多个实例同时跑，会不会重复推送？

**A：** 理论上存在竞态窗口——两个实例同时检查 pushedKey 都不存在，然后同时发送 MQ。当前代码的 pushedKey 是发送成功后才设置的，属于 check-then-act 模式。

实际场景中这个概率极低（5 分钟一次，窗口在毫秒级），但如果要彻底解决，可以改为发送前先 SETNX 抢占 pushedKey，利用 Redis 的原子性保证只有一个实例能抢到。这也是后续优化的方向。

---

### Q25：RabbitMQ 的重试策略是怎么配置的？

**A：** 在 `application.yaml` 中配置：
```yaml
listener.simple.retry:
  enabled: true
  max-attempts: 3        # 最多3次
  initial-interval: 2000  # 首次重试等2秒
  multiplier: 2           # 指数退避倍乘
  max-interval: 10000     # 最大间隔10秒
default-requeue-rejected: false  # 重试耗尽不进原队列，进DLX
```

重试间隔：2s → 4s → 8s，总共最多等 14 秒。3 次全部失败后，`default-requeue-rejected: false` 确保消息不会无限循环，而是路由到死信队列。

---

### Q26：死信队列怎么配置的？消息进入死信后怎么办？

**A：** 在 `RabbitMQConfig` 中声明主队列时，通过 `x-dead-letter-exchange` 和 `x-dead-letter-routing-key` 参数绑定死信交换器。重试耗尽后，RabbitMQ 自动将消息投递到 `reim.dlx.exchange` → `reim.dlx.queue`。

`BpmMessageListener.handleDeadLetterMessage()` 消费死信，记录严重告警日志，并通过钉钉通知管理员："报销单 XXX 推送 BPM 失败，重试 3 次已耗尽，请立即人工介入"。

---

### Q27：如果整个 RabbitMQ 服务宕机了怎么办？

**A：** 
1. 用户提交报销单不受影响——MQ 发送失败被 catch，消息表 status 标记为 2（发送失败），业务正常返回成功
2. 补偿任务每 5 分钟扫描，发现未推送的单据会不断重试
3. MQ 恢复后，补偿任务下一次扫描时自动补发
4. 如果长时间未恢复，熔断器触发钉钉告警，管理员可以手动介入或等到 MQ 恢复后自动恢复

---

## 七、@Idempotent 幂等性（亮点四）

### Q28：@Idempotent 注解的实现原理是什么？

**A：** 基于 Redis + Lua 脚本的 AOP 拦截器。

核心是 `IdempotentAspect` 环绕通知：
1. 从请求中提取 userId（Header 或远程 IP）
2. 构造幂等 key：`idempotent:{接口URI}:{userId}:{参数MD5}`
3. 执行 Lua 脚本：
   ```lua
   if redis.call('setnx', KEYS[1], ARGV[1]) == 1 then
       redis.call('expire', KEYS[1], ARGV[2])
       return 1   -- 首次请求，放行
   else
       return 0   -- 重复请求，拒绝
   end
   ```
4. 返回 1 → 放行；返回 0 → 抛出 BusinessException("操作太频繁")

关键设计：Lua 脚本保证了 SETNX + EXPIRE 的原子性，不会出现"设了 key 但没设过期时间导致死锁"的情况。

---

### Q29：幂等 key 为什么用 MD5？会不会有哈希冲突？

**A：** 用 MD5 是为了压缩 key 长度。如果请求参数很大（比如包含长 JSON），直接拼接会导致 Redis key 过长，影响性能和内存。MD5 固定 32 位。

MD5 的碰撞概率在 128 位空间下极低（2^-128），在实际业务场景中可以忽略不计。而且即使理论上有碰撞，同一个用户在 5 秒窗口内、同一个接口下碰撞的概率更是天文数字。如果真的担心，可以换成 SHA-256 或者直接保留原始参数。

---

### Q30：如果 Redis 挂了，@Idempotent 会怎样？

**A：** 这是一个需要诚实回答的问题。当前代码中，如果 `stringRedisTemplate.execute()` 抛异常，没有 try-catch，异常会直接向上传播，最终被 `GlobalExceptionHandler` 兜底，返回 500 系统错误。

也就是说，**当前是 fail-close 模式**——Redis 不可用时，所有带 `@Idempotent` 的接口都不可用。如果业务要求高可用，应该改成 fail-open（Redis 异常时放行，只记日志），牺牲幂等性保证可用性。这是一个需要根据业务场景权衡的设计决策。

---

### Q31：@Idempotent 的 5 秒超时是怎么定的？为什么不是 1 秒或 10 秒？

**A：** 5 秒是一个经验值。太短（1 秒）可能不够——用户网络延迟 + 前端防抖失效时，两次请求可能间隔 2-3 秒；太长（10 秒）会影响正常的重复操作——用户如果真的是想修改后重新提交，要等 10 秒。5 秒是一个平衡点。

这个值可以通过注解参数自定义：`@Idempotent(timeout = 10)`，不同业务场景可以设不同的窗口。

---

## 八、Redisson 分布式锁

### Q32：Redisson 锁和 @Idempotent 有什么区别？为什么两个都要？

**A：** 两者的保护维度不同：

| | @Idempotent | Redisson 锁 |
|---|-------------|-------------|
| 保护范围 | 同一用户 + 同一参数 | 同一报销单 ID |
| 技术实现 | Redis SETNX + Lua | Redisson RLock |
| 防护目标 | 防用户重复点击 | 防集群内并发提交 |
| 维度 | 用户级 | 单据级 |

举例说明区别：
- 用户 A 在 1 秒内点两次提交按钮 → `@Idempotent` 拦截（相同用户+相同参数）
- 用户 A 和 用户 B 同时分别提交同一份报销单 → `@Idempotent` 放行（不同用户），但 Redisson 锁拦截（同一 reimId）

---

### Q33：tryLock(0, 10, TimeUnit.SECONDS) 三个参数分别是什么意思？

**A：**
- `waitTime = 0`：拿不到锁立即返回 false，不排队等待。如果有人在提交，后来者直接提示"请勿频繁重复提交"
- `leaseTime = 10`：锁的租约 10 秒。如果业务逻辑超过 10 秒锁自动释放，防止死锁
- `TimeUnit.SECONDS`：时间单位

为什么 waitTime=0？因为提交是互斥操作，同一份报销单同时被两个人提交本身就是异常情况，不需要排队，直接拒绝更合理。如果业务需要排队（比如秒杀），才会设 waitTime > 0。

---

### Q34：Redisson 锁怎么防止死锁？

**A：** Redisson 的 RLock 有看门狗机制——如果 `leaseTime` 没指定（-1），Redisson 会自动每 30 秒续期一次，只要持有锁的节点还活着，锁就不会过期。

但我这里指定了 `leaseTime=10`，明确给了租约时间，即使业务逻辑卡死，10 秒后锁自动释放，不会死锁。这是主动防御策略。

---

### Q35：锁的 key 为什么是 `fk:reim:lock:submit:{reimId}`？

**A：** 锁的粒度是**单号级别**。不同报销单的提交互不影响，只有同一个报销单的并发提交才互斥。如果用全局锁（比如 `fk:reim:lock:submit`），会导致所有报销单的提交都串行化，性能会严重下降。

## 九、三层防护体系

### Q36：你把幂等性分了三层，能完整讲一下三层分别解决什么问题吗？

**A：** 三层防护从外到内：**注解 → 分布式锁 → 乐观锁**，分别解决不同维度的问题：

| 层级 | 技术 | 解决的问题 | 失效场景 |
|------|------|-----------|---------|
| 第一层 | @Idempotent + Redis Lua | 同一用户在 5s 内重复提交相同请求 | 不同用户提交相同单据 |
| 第二层 | Redisson RLock | 同一 reimId 在集群中的并发提交 | 锁过期、Redis 不可用 |
| 第三层 | @Version 乐观锁 | 数据库级别的写冲突 | 不传 version 的更新操作 |

三层互补，即使某一层失效，其他层仍能兜底。比如 Redis 宕机导致 @Idempotent 和 Redisson 都失效，数据库的乐观锁仍然能防止数据被覆盖。

## 十、综合与场景题

### Q37：从用户点击"提交"到报销单状态变为"已完成"，完整的调用链路是怎样的？

**A：**
1. 前端发起 `POST /fccapi/REIM_Submit`
2. `AuthInterceptor` → 提取 userId，设置 UserContext
3. `IdempotentAspect` → 检查 Redis，通过则放行
4. `ReimMainController.submitReim()` → 进入 Service
5. `ReimMainServiceImpl.submitReim()`：
   - Redisson 获取分布式锁
   - 校验状态、版本、完整性
   - 更新 reimStatus 为 1（@Version 乐观锁保护）
   - 事务内：插入 `fk_mq_message`（status=0）
   - 事务外：RabbitTemplate 发送到 `reim.submit.topic`
   - 成功 → 消息表 status=1；失败 → status=2
   - finally：释放 Redisson 锁
6. `ReimLogAspect` → 环绕通知获取前后快照，比对状态变化，写入 `fk_reim_log`
7. `AuthInterceptor.afterCompletion()` → clear UserContext
8. 返回成功响应

---

### Q38：如果用户在提交过程中关闭了浏览器，报销单会处于什么状态？

**A：** 取决于请求在服务端执行到哪一步：
- 如果还没到数据库更新：报销单仍是草稿，可以重新提交
- 如果数据库已更新但 MQ 还没发：状态为 1（已完成），补偿任务会在 5 分钟后补发 MQ
- 如果全部完成但响应没到前端：状态为 1，MQ 已发，前端超时——用户刷新页面即可看到最新状态

---

### Q39：如果 BPM 审批流回调超时或失败怎么办？

**A：** BPM 回调 `REIM_UpdateStatus` 接口通过 `SignAuthInterceptor` 签名验证。如果回调失败，BPM 侧应有自己的重试机制。如果 BPM 不重试，我们的补偿任务也能间接发现问题——报销单长期处于已推送但状态未更新的状态，可以通过监控 `fk_mq_message` 表或 `update_time` 来告警。

这是我们需要和 BPM 团队约定好的 SLA 对接点。

---

### Q40：你觉得自己实现的这些功能里，哪个最有技术含量？为什么？

**A：** 我认为**本地消息表 + 可靠消息投递**方案最有技术含量。不是因为它代码最复杂，而是因为它解决的是一个真实的分布式系统难题——跨服务的数据一致性。

很多同学可能直接用 `@Transactional` + MQ 发送，但那样在 MQ 发送失败时没有补偿机制。我通过学习业界成熟方案（如 RocketMQ 的事务消息、RocketMQ 的半消息机制），用"本地消息表"这个经典模式在 RabbitMQ 上实现了最终一致性。从数据库事务保证原子记录，到补偿任务的定时扫描，到死信队列的人工兜底，形成了一个完整的闭环。

这是从"能跑就行"到"生产可用"的思维转变，我认为这比单纯的 CRUD 更能体现一个后端开发者的架构意识。

### Q41：如果让你重新设计这个系统，你会做哪些改进？

**A：**
1. **状态机正规化**：引入状态枚举 + 状态机引擎（如 Spring State Machine 或 Cola 状态机），替代当前的魔数 + if-else
2. **消息表方案升级**：考虑直接用 RocketMQ 的事务消息，省去本地消息表和补偿任务的维护成本
3. **Redis 不可用降级**：@Idempotent 增加 fail-open 开关，允许配置为 Redis 异常时放行
4. **日志敏感信息脱敏**：AOP 日志中的报销人姓名等信息走统一的脱敏逻辑
5. **补偿任务分布式锁**：防止多实例下的并发补偿
6. **cancelReim 加强乐观锁**：作废操作也应该接收 version 参数
7. **API 文档自动化**：接入 Swagger/Knife4j，减少手动维护接口文档的成本
