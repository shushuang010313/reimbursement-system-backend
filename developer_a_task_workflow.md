# 开发者A (舒爽) 核心任务与数据流向梳理

本文档详细梳理了开发者A负责的核心业务（主流程、费用分摊）及高阶架构（MQ最终一致性、接口幂等性防抖、AOP日志等）在系统中的完整流转过程。以最核心的“保存”和“提交”两个场景为例，从前端传参一路追踪到底层数据库和中间件，方便在联调和项目答辩时做到逻辑闭环。

## 场景一：保存/修改报销单主数据及费用分摊

**核心目标**：完成报销单草稿的保存，同步完成复杂费用分摊的试算，防范并发覆盖，并由底层自动捕获异动日志。

### 1. 前端传参与网关层
- **前端动作**：用户填写完报销单头部基础信息，并配置了第 2~N 行的费用分摊比例后，点击“保存”按钮。
- **请求参数模型 (`ReimSaveDTO`)**：
  - `id` (若为修改时必传)
  - `version` (当前前端持有的乐观锁版本号)
  - **基础信息**：`reimbursementTitle` (报销标题), `businessTripReason` (出差事由), `reimCompanyId` 等。
  - **分摊列表 (`splitList`)**：包含多行分摊明细数据（注意：前端不应传第 1 行的比例，或者即使传了后端也会强制重算）。
- **网关/全局拦截器**：请求进入系统后，首先经过全局 `AuthInterceptor`，解析 HTTP Header 中的 Token，并将当前操作的用户信息（如 UserId、UserName）安全地存入 `UserContext` 线程上下文中。

### 2. 控制器层 (`ReimMainController`)
- **接口入口**：`@PostMapping("/fccapi/REIM_Save")`
- **操作逻辑**：接收被 `@RequestBody` 包装的 `ReimSaveDTO`，执行基本的数据非空校验（如 JSR303 `@Validated`），随后调用 `IReimMainService.saveReimData()`。

### 3. 业务服务层与算法计算 (`ReimMainServiceImpl`)
- **前置并发校验**：
  - 判断单据库中状态，若已是“已完成”或“审批中”，则阻断修改。
  - 若是修改操作，对比 DTO 传入的 `version` 与数据库库中现存的 `version`。如果不一致，抛出“数据已被他人修改”的乐观锁异常。
- **用户数据提取**：无需前端传操作人 ID，直接通过 `UserContext.getUserId()` 提取安全的用户身份，自动填充表实体中的 `createUserId` 或 `updateUserId`。
- **🌟 调用分摊引擎 (倒挤法核心算法)**：
  - 将 `splitList` 传递给内部的分摊计算核心方法。
  - **倒挤法执行**：遍历分摊列表，累加第 2 行到第 N 行的 `splitRatio`。
  - **防超限校验**：若 `SUM(第2~N行比例) > 100%`，立即抛出业务异常 `REIM_004` (分摊比例超过100%)。
  - **计算第一行兜底**：第一行分摊比例严格等于 `100% - SUM(其他行比例)`。
  - **金额分配精度控制**：其他行分摊金额 = `单行比例 * 主单补助总金额 (subsidyTotal)`；第一行分摊金额 = `总金额 - SUM(其他行分摊金额)`。（绝对防范 `0.01` 元的浮点数精度丢失问题）。
- **持久化操作 (`MyBatis-Plus`)**：
  - 在 `@Transactional` 事务边界内，执行 `reimMainMapper.insert/updateById`。
  - 若存在历史分摊数据则先清理，随后调用 `reimSplitMapper.insertBatch` 批量落库最新的分摊明细。

### 4. 切面与底座拦截 (`AOP`)
- **自动异动日志抓取 (`ReimLogAspect`)**：
  - 切面静默捕获到 `update` 数据库动作，利用 Java 反射机制自动对比修改前后的字段旧值与新值。
  - 生成格式化的追踪日志（如：“【报销事由】从 [市场调研] 修改为 [拜访客户]”），并与业务同事务（或异步化）持久化进入 `fk_reim_log` 表，形成审计凭证。

## 场景二：提交报销单 (涉及高并发防抖与 MQ 高可靠)

**核心目标**：将报销单状态扭转为“已完成/审批中”，并通过 MQ 异步推送到下游的 BPM 工作流系统。要求**绝对防抖（不生成重复单据）**且保证消息在分布式环境下 **100% 不丢失**。

### 1. 前端传参
- **前端动作**：用户确认所有行程、补助、分摊无误后，在详情页点击“提交”按钮。
- **请求参数模型 (`ReimSubmitDTO`)**：
  - `reimId` (报销单主键)
  - `version` (当前单据版本号)

### 2. 🌟 拦截器层：高并发防重发 (亮点四：绝对幂等)
- **幂等拦截入口 (`IdempotentAspect`)**：
  - 在 `ReimMainController.submit()` 接口方法上，标记了 `@Idempotent(key="#dto.reimId", timeout=3)` 注解。
  - 请求到达前，AOP 切面基于传入的 `reimId` 生成全局唯一的 Redis Key。
  - **Lua 脚本原子操作**：在 Redis 中执行包含 `SETNX` 语义的 Lua 脚本，若发现该 Key 存在，说明该用户在 3 秒内对同一单据发生了极其频繁的连击，拦截器直接抛出“请求过于频繁，请勿重复提交”的提示。在 Web 接入层就拦截了恶意并发，保护了底层的数据库连接池。

### 3. 控制器层与分布式锁 (`ReimMainController` -> `ReimMainServiceImpl`)
- **接口入口**：`@PostMapping("/fccapi/REIM_Submit")`
- **Redisson 分布式锁加固**：
  - 进入 Service 后，首先尝试获取基于 Redisson 的分布式业务锁 `fk:reim:lock:submit:{reimId}`，确保在集群部署模式下，同一张单据的提交请求只会由一个节点的一个线程处理，双重防重。
- **状态流转与乐观锁更新**：
  - 检查单据完整性（必须有行程明细、金额不能为0等空单据拦截）。
  - 携带 `version` 进行数据库条件 Update：`UPDATE fk_reim_main SET reim_status = 1, version = version + 1 WHERE id = ? AND version = ?`。若影响行数为 0，说明被他人抢先操作，抛出并发修改异常。

### 4. 🌟 MQ 可靠消息投递构建 (亮点三：分布式一致性)
- **构建本地消息追踪表 (`fk_mq_message`)**：
  - 组装需要推送给 BPM 系统的复杂报销单据 Payload（金额、报销人、事由等 JSON）。
  - **核心设计**：在当前提交业务的 `@Transactional` 内部，向 `fk_mq_message` 表插入一条记录，状态标记为 `0` (待发送)。这保证了“单据状态更新为1”和“待发送消息入库”在同一个本地事务中，**同成功、同回滚**。
- **触发 MQ 发送与 ACK**：
  - 事务提交后，调用 `RabbitTemplate` 将消息异步投递到 `reim.submit.topic`。
  - **ConfirmCallback 机制**：服务端接收 RabbitMQ Broker 返回的 ACK 回执。如果明确收到成功回执，则将 `fk_mq_message` 表中的那条记录状态更新为 `1` (发送成功)。

### 5. 异常兜底与死信队列机制 (DLX)
- **网络抖动/发送失败兜底**：
  - 若 `RabbitTemplate` 发送超时、网络闪断，导致未收到 ACK，此时 `fk_mq_message` 中的状态依然锁定为 `0`。
  - **补偿任务 (`CompensationJob`)**：定时任务系统每隔几分钟扫描表中长期处于 `0` 状态的消息，重新拉起投递逻辑。采用指数退避策略（如失败1次等1分钟再试，再失败等5分钟）。
- **消费端异常与死信队列 (DLX) 防丢**：
  - 如果消息成功进入了 RabbitMQ，但 BPM 消费端因为宕机、代码 Bug 导致消费抛错。消息在触发 3 次消费重试后，会被自动路由到绑定的 `Dead Letter Exchange (死信交换机)`。
  - 系统配置了专门的死信消费者，一旦监听到死信，立即截获异常日志并触发飞书/钉钉的运维告警机制。交由人工介入（修改错误数据后重新放回正常队列），形成分布式闭环。
