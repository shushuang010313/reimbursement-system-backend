# Git 提交规范 (Commit Convention)

【基本格式】 类型: 提交的具体内容描述 (注意：冒号后面必须有一个空格)

【支持的“类型”字典】 请根据你提交的代码性质，选择以下前缀：

- 🚀 feat: 新增功能（Feature），比如：写了一个新的接口、建了新的 Service。
  示例：`feat: 新增获取补助日历列表接口`
- 🐛 fix: 修复 Bug，比如：修复了测试提出的缺陷或线上报错。
  示例：`fix: 修复行程时间重叠校验失效的问题`
- 🔨 refactor: 重构代码（即不是新增功能，也不是修改bug的代码变动，比如优化了一段算法）。
  示例：`refactor: 优化补助金额计算引擎的执行效率`
- 📝 docs: 仅仅修改了文档（如 README、设计文档等）。
  示例：`docs: 更新差旅报销单详细设计文档`
- 🎉 init: 项目初始化（通常只在搭建脚手架或大模块初创时使用）。
  示例：`init: 初始化后端基础项目架构，包含实体类与公共拦截器`
- ⚙️ chore: 构建过程、辅助工具、配置文件的变动（比如修改了 pom.xml 依赖）。
  示例：`chore: 升级 mybatis-plus 依赖版本到 3.5.3`
