# 费控云差旅报销系统 - 核心功能与设计学习指南

基于你的要求，我们将完全跳过**开发者 B（行程、补助明细、日历引擎）**的相关工作，专注于系统主干流程、费用分摊以及底层的中间件对接设计。

为了让你更容易理解和吸收，我们将这些功能模块从易到难分为三个阶段进行学习：

---

## 🟢 第一阶段：基础业务与报销单生命周期（简单）

这部分是整个系统的骨架，负责承载用户最基本的操作。

### 1. 报销主单的分页查询 (`ReimMainServiceImpl.queryPageList`)
* **设计目的**：在工作台展示用户的报销单列表，支持多种维度的搜索。
* **核心内容**：
  * 使用了 MyBatis-Plus 的动态 SQL（`<where>` 标签）。
  * 实现了基于报销单号 (`reimNo`)、标题 (`reimTitle`)、事由 (`businessTripReason`) 的模糊查询（`LIKE`），以及公司/部门的精确匹配。
  * **数据格式化**：在后端对金额字段进行了拦截处理，强制保留两位小数，然后封装为 `Page<ReimMainVO>` 结构返回给前端。

### 2. 保存草稿与单号生成 (`ReimMainServiceImpl.saveReimMain`)
* **设计目的**：用户在填写报销单中途可以保存进度。
* **核心内容**：
  * **自动编码规则**：系统会自动生成业务单号，规则为：`REIM` + 年月日 + 流水号（例如：`REIM202605270001`）。
  * **乐观锁防并发**：这是并发控制的第一层。数据库表和 MyBatis-Plus 实体类中使用了 `@Version` 字段。如果两人同时编辑同一份草稿，第二个人保存时版本号对不上，系统会自动拦截并抛出提示，防止数据相互覆盖。

## 🟡 第二阶段：复杂计算逻辑与高并发保护（中等）

在单据流转过程中，涉及到财务分摊和敏感的状态变更，这一阶段引入了更复杂的算法和锁机制。

### 1. 费用分摊联动计算 (`ReimSplitServiceImpl.calculateSplitRatio`)
* **业务场景**：一笔出差费用可能需要由多个不同的部门或项目共同承担（例如 A 部门承担 30%，B 部门承担 70%）。
* **核心算法设计**：
  * **反向推导机制**：为了保证分摊比例永远是 100%，系统规定**第一行的分摊比例 = 100% - Σ(第二行及后续所有行的比例之和)**。
  * **严格拦截**：如果后端检测到所有分摊比例之和超过 100%，会立即中断并抛出 `REIM_004` 业务异常。
* **财务级精度控制**：
  * 因为涉及到钱的计算，绝不能使用 `Double` 或 `Float`。代码中强制使用 `BigDecimal` 进行乘法运算，并结合 `RoundingMode.HALF_UP`（四舍五入）来处理小数点，彻底杜绝了财务计算中的精度丢失问题。

### 2. 报销单提交与分布式防重 (`ReimMainServiceImpl.submitReim`)
* **业务场景**：用户填完报销单点击“提交”。此时如果网络卡顿，用户疯狂连击“提交”按钮，会导致系统重复发起多笔相同的审批流。
* **分布式锁设计**：
  * 引入了 `Redisson` 分布式锁。当用户点击提交时，后端以单据 ID 为 Key（例如：`fk:reim:lock:submit:{reimId}`）去 Redis 申请一把锁。
  * 拿到锁的线程正常往下走；拿不到锁的请求（连击产生的多余请求）会被直接拒绝。
* **核心业务拦截**：在真正流转状态前，进行**兜底校验**（必须有关联的行程、总金额必须大于0），校验通过后，将单据状态 `reimStatus` 修改为 `1 (已完成/审批中)`。

## 🔴 第三阶段：底层基础设施与第三方系统对接（较难）

这部分属于高级架构设计，目的是为了保证系统的可维护性、数据安全性以及与审批系统的最终一致性。

### 1. 异动日志自动追踪 (`ReimLogAspect`)
* **设计目的**：审计需求。系统需要知道“谁在什么时间，把报销单的哪个字段从 A 改成了 B”。
* **AOP 核心设计**：
  * 没有把记日志的代码硬塞在业务代码里，而是使用了 Spring AOP（面向切面编程）的 `@Around` 或 `@AfterReturning` 注解。
  * 只要单据状态发生变更的方法被调用，切面就会自动拦截，对比内存中修改前和修改后的实体数据。
  * 生成日志后持久化到 `fk_reim_log` 表中。同时，如果内容包含银行卡号、身份证等敏感字段，还会进行自动打码（如 `6222****1234`）。

### 2. 审批流对接与 MQ 补偿机制 (`BpmCompensationJob`)
* **业务场景**：单据提交后，需要去公司的 OA 系统/BPM 系统发起真正的审批流。
* **MQ 异步解耦**：
  * 提交成功后，业务系统不直接调 OA 接口等待，而是向 RabbitMQ 的 Topic（`reim.submit.topic`）发送一条消息。消费者收到消息后再去慢慢调 OA 接口，这样极大提升了用户点击提交时的响应速度。
* **XXL-Job 极致容灾（分布式事务补偿）**：
  * **痛点**：如果给 RabbitMQ 发消息失败了，或者 RabbitMQ 挂了怎么办？报销单岂不是永远卡在“待推送”状态？
  * **解决方案**：引入 XXL-Job 定时任务。系统每隔一段时间会自动去数据库扫描状态为 `待推送` 且更新时间在 5 分钟之前的数据。
  * 发现这种“遗漏”数据，系统会自动重新尝试投递。如果连续重试超过 3 次依然失败，系统会触发告警通知开发人员介入。这种机制确保了分布式系统间的**最终一致性**。
