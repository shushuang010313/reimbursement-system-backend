## 开发须知

详细设计是在充分理解需求的基础上，进行设计文档的编写，设计文档应充分说明复杂功能、核心功能关键实现方法、步骤、路径，体现设计者对于需求的理解，以及知道如何实现。

设计文档也是指导实现者如何实现的参考指南。详细设计不仅要关注功能性需求的设计，同时也包含非功能性需求的设计。

### 1. 非功能性要求

#### 可靠性

- **容错**：系统在发生故障时仍保持正常运行，分布式系统发生通信异常时会先暂时切断连接，故障修复后重新连接恢复运行
- **健壮性**：系统能抵御非正常使用方式或非法输入，系统缺陷率每1000小时最多发生1次故障
- 因软件系统失效而造成无法完成业务的概率小于5‰

#### 高性能

- **响应时间**：95%情况下，一般时段响应时间不超过1.5秒，高峰时段不超过4秒
- **并发数**：系统可同时满足10,000个用户请求
- **资源使用率**：CPU占用率 ≤ 50%，内存占用率 ≤ 50%

#### 可维护性

- **普通修改**：接到修改请求后，1~2天内完成
- **重大需求/设计修改**：评估后1周内完成
- 90% BUG修改时间不超过1个工作日，其他不超过2个工作日

#### 安全性

- 严格权限访问控制，用户经身份认证后，只能访问权限范围内的数据和操作
- 提供运行日志管理及安全审计功能，可追踪系统历史使用情况
- 能经受来自互联网的一般性恶意攻击：病毒、口令猜测攻击、黑客入侵等
- A公司员工不能访问B公司的数据，数据隔离严格执行

### 2. 详细设计评审要求

- 详细设计文档完成后需召开评审会议，参与人员包括：产品经理、技术架构师、前后端开发负责人、测试负责人
- 评审重点：业务逻辑完整性、接口定义合理性、数据库设计规范性、异常处理方案完备性
- 评审通过后方可进入编码阶段

## 2. 功能性需求描述

### 2.1 需求概述

#### 2.1.1 业务背景

员工因公出差，或为企业办理业务，自己先垫钱消费，业务结算后与企业进行报账结算；也有员工提前向公司借款，办理业务完成后进行多退少补的结算。如果员工出差全部走线上预订，由公司/服务商/供应商垫付资金，员工出差回来后拿相关票据报账，此时报销金额为0，企业财务对服务商/供应商进行对账结算。

#### 2.1.2 需要解决的财务痛点

- **审批耗时耗力**：员工携带贴票报销单逐级审批，财务人员审核时需考虑多维因素，人力与时间成本高
- **业务真实性难以保障**：消费时间真实性（行程是否重复）、消费金额真实性（是否与发票金额一致）、费用真实性（是否有无关发票）
- **票据合规性难以验证**：查真查重、发票抬头与税号一致性、连号检测、发票有效期检查
- **财务与员工矛盾**：人工审批时间慢，票据问题多，打回重提单据频繁，员工体验差

#### 2.1.3 需求概述

- **报销单列表功能**：支持多条件查询（报销单号、标题、事由、费用归属公司、报销部门、报销人、业务类型）及列表展示
- **报销单据页面**：包含单据头部、基本信息、补录行程、补助信息、补助日历、费用合计、费用分摊、备注信息等功能分区
- **补助自动计算**：根据出差城市等级自动匹配餐费/交通/通讯补助标准，支持日历维度勾选管理
- **费用分摊管理**：支持多行费用分摊，第一行比例自动联动计算，确保总比例100%
- **单据状态管理**：草稿（0）→ 已完成（1）→ 已作废（2）

### 2.2 业务全景图

差旅报销单业务全景：

| 环节 | 操作人 | 主要动作 | 系统响应 |
| --- | --- | --- | --- |
| 发起报销 | 员工 | 新增报销单，填写基本信息 | 生成草稿状态报销单 |
| 补录行程 | 员工 | 录入出差行程（出行人、出发/到达城市、日期） | 生成补助信息及补助日历数据 |
| 确认补助 | 员工 | 在补助日历中勾选实际补助项，修改金额 | 实时计算补助金额合计 |
| 费用分摊 | 员工 | 配置费用归属公司和分摊比例 | 自动计算第一行分摊比例 |
| 提交报销 | 员工 | 所有信息填写完整后提交 | 单据状态变更为已完成 |
| 财务审核 | 财务人员 | 审核单据真实性、合规性 | 审批通过或打回 |
| 结算报销 | 财务人员 | 完成报销金额结算 | 报销流程结束 |

## 3. 功能详细设计

### 3.1 报销单列表

#### 3.1.1 功能内容

提供报销单的多条件查询和列表展示功能，支持新增、搜索、跳转详情等操作。

#### 3.1.2 查询条件设计

| 查询条件 | 输入方式 | 数据来源/说明 |
| --- | --- | --- |
| 报销单号 | 手动输入 | 模糊匹配 |
| 标题 | 手动输入 | 模糊匹配报销标题 |
| 事由 | 手动输入 | 模糊匹配出差事由 |
| 费用归属公司 | 下拉选择 | 调用公司列表接口，返回法人公司列表 |
| 报销部门 | 下拉选择 | 调用部门列表接口，支持部门树形结构 |
| 报销人 | 下拉选择 | 调用员工列表接口，展示姓名+工号 |
| 业务类型 | 树形下拉选择 | 调用业务类型接口，支持多级树形结构 |

#### 3.1.3 列表展示字段

| 字段名称 | 展示要求 | 说明 |
| --- | --- | --- |
| 操作 | 操作列 | 超过三个操作时第三个展示[更多]图标；鼠标移入展示其他操作 |
| 报销单号 | 展示主题色，可点击 | 点击单号跳转单据详情页面 |
| 单据状态 | 状态标签 | 草稿/已完成/已作废，不同颜色区分 |
| 报销人 | 员工姓名+工号 | 格式：张三（10001） |
| 报销部门 | 部门名称+编号 | 格式：财务部（D001） |
| 费用归属公司 | 公司名称 | 展示法人公司全称 |
| 业务类型 | 类型名称 | 展示末级业务类型名称 |
| 报销标题 | 展示主题色，可点击 | 点击标题跳转单据详情页面 |
| 报销事由 | 文本 | 超长省略号显示，鼠标悬浮完整展示 |
| 补助金额 | 金额右对齐 | 保留两位小数，带千分位 |
| 创建时间 | 日期时间 | 格式：yyyy-MM-dd HH:mm:ss |

#### 3.1.4 操作按钮逻辑

| 按钮 | 操作逻辑 |
| --- | --- |
| 新增 | 点击跳转报销单详情页面（新增模式） |
| 搜索 | 根据查询条件调用后端查询接口，刷新列表数据 |
| 清除 | 清空所有查询条件输入框内容 |
| 编辑 | 列表行操作，跳转到报销单详情页面（编辑模式） |
| 作废 | 弹出确认对话框，确认后将单据状态置为已作废（code:2） |
| 删除 | 草稿状态下可删除，弹出确认后物理删除 |

#### 3.1.5 实现逻辑

列表查询流程：

- 前端组装查询参数（current, size, 各查询条件），调用分页查询接口
- 后端通过MyBatis动态SQL拼接查询条件，支持模糊查询和精确匹配
- 返回分页数据，包含records列表和total总条数
- 前端根据返回数据渲染列表，金额字段右对齐显示

#### 3.1.6 异常处置

- **网络超时**：展示加载失败提示，提供重试按钮
- **查询无结果**：展示空状态图，提示[暂无数据]
- **接口异常**：捕获异常后弹出错误提示，日志记录异常信息

### 3.2 报销单据页面

#### 3.2.1 功能内容

报销单据页面是差旅报销单的核心功能页面，包含8个功能分区：单据头部、基本信息、补录行程、补助信息、费用合计、费用分摊、备注信息、操作按钮。

页面规格：内容宽度1200px，单据标题20px加粗，分区标题16px，其他字段14px，分区标题高度36px。

#### 3.2.2 单据头部

- 表头固定在单据最上面，不随页面纵向滚动条滚动（position: sticky; top: 0）
- 单据标题居中展示，日期靠右对齐
- 日期取值：新增时取当前日期；编辑时取页面保存的数据日期
- 单据状态展示在标题旁（草稿/已完成/已作废）

#### 3.2.3 基本信息分区

- 分区头部：点击箭头图标实现展开/收起效果

**字段说明：**

| 字段名 | 输入方式 | 验证规则 | 说明 |
| --- | --- | --- | --- |
| 报销标题 | 手动输入 | 必填，≤500字 | |
| 出差事由 | 手动输入 | 必填，≤500字 | |
| 报销人 | 下拉选择 | 必填 | 调用员工控件接口 |
| 报销部门 | 下拉选择 | 必填 | 联动报销人自动带出所属部门，可修改 |
| 费用归属公司 | 下拉选择 | 必填 | 调用法人公司接口 |

#### 3.2.4 补录行程分区

- 点击【补录行程】按钮，弹框展示补录行程页面
- 列表展示字段：出行人员、出差日期、行程（出发城市-到达城市）、行程说明、操作列
- 操作列功能：删除、编辑、复制

**弹框字段逻辑：**

| 字段 | 输入方式 | 验证规则 |
| --- | --- | --- |
| 出行人 | 下拉选择 | 必填，支持多选 |
| 出发城市 | 下拉选择 | 必填 |
| 到达城市 | 下拉选择 | 必填 |
| 出发日期 | 日期选择 | 必填，不可晚于当前日期 |
| 到达日期 | 日期选择 | 必填，不可早于出发日期，不可晚于当前日期 |
| 行程说明 | 手动输入 | 必填，≤500字 |

- **跨天跨城行程说明**：出发城市-到达城市：武汉-北京；出发日期-到达日期：1号-5号；1号~5号补助按北京匹配
- 保存时前后端均需校验：必填字段、人员+日期范围不可重复
- 保存后自动生成补助信息数据及对应补助日历数据
- **编辑操作**：弹框回显已选择的数据
- **删除操作**：弹出确认提示框，确认后删除并级联删除对应补助信息
- **复制操作**：弹框展示复制数据，保存后生成新的一条数据（需校验行程时间不重复）

#### 3.2.5 补助信息分区

- 分区头部：点击图标实现展开/收起效果
- 头部展示补助提示信息，超出使用省略号，鼠标移入悬浮完整显示

**提示信息内容**：「1、请根据实际出差日期选择补助；2、出差期间当日有用餐安排的请自行核减当日餐补；3、出差期间当日有用车的，请自行核减当日交补」

**列表展示字段：**

| 字段 | 说明 |
| --- | --- |
| 出行人 | 补录行程中的出行人员 |
| 出差日期 | 补录行程中的出差日期 |
| 补助天数 | 出差日期对应的天数 |
| 行程 | 补录行程中的出发城市-到达城市 |
| 补助城市 | 补录行程中的到达城市 |
| 申请金额 | 补助日历中勾选项的标准金额合计 |
| 补助金额 | 补助日历中勾选项的实际补助金额合计 |
| 操作 | 编辑按钮（弹出补助日历页面） |

#### 3.2.6 补助日历弹窗

- **出差类型**：与报销单主单上的业务类型展示一致
- **开始/结束日期**：取补录行程分区中的出发日期、到达日期
- **行程天数地点**：取补录行程中出发/到达城市，天数取开始到结束之间的天数

**补助标准**（城市等级由城市数据csfllx字段决定）：

| 补助类型 | 一线城市 | 二线城市 | 三线城市 |
| --- | --- | --- | --- |
| 餐费补助 | 100元/天 | 80元/天 | 50元/天 |
| 交通补助 | 40元/天 | 40元/天 | 40元/天 |
| 通讯补助 | 40元/天 | 40元/天 | 40元/天 |

**补助日历交互逻辑：**

- 补助日历数据根据出发日期到到达日期之间的天数生成每条记录
- 每条数据需根据日期获取对应的星期数
- 补助城市取补录行程中的到达城市
- **出差日期的复选框**：实现横向列选中功能（选中某天的全部补助项）
- **补助项复选框**：实现纵向列选中功能（选中某补助项的全部天数）
- **表头复选框**：全选/全取消所有复选框
- 只有当前行或当前列中所有复选框都选中，对应表头/出差日期的复选框才是选中状态
- 未选中的数据，输入框不可编辑
- 输入框只能输入正数，不可大于标准金额
- **申请金额**：取补助日历中的标准金额值
- **补助金额**：取补助日历中的实际填写金额值

#### 3.2.7 费用合计分区

- 分区头部：点击图标实现展开/收起效果
- **补助总金额** = 补助信息中所有记录的补助金额之和
- **餐费补助** = 补助信息中每条记录中餐费补助金额之和
- **交通补助** = 补助信息中每条记录中交通补助金额之和
- **通讯补助** = 补助信息中每条记录中通讯补助金额之和

#### 3.2.8 费用归属及分摊分区

- 分区头部：显示分摊金额 = 费用合计中的补助总金额

**字段说明：**

| 字段 | 输入方式 | 规则说明 |
| --- | --- | --- |
| 费用归属 | 下拉选择 | 调用法人公司接口 |
| 项目 | 下拉选择 | 调用项目接口 |
| 分摊比例 | 数字输入 | 第1行不可编辑，自动计算；第2+行手动输入 |
| 分摊金额 | 自动计算 | = 分摊比例 × 补助总金额 |

- **均摊按钮**：根据分摊数据条数对比例和金额进行平均分摊
- **添加一行**：新增分摊数据，默认费用归属和项目为空，分摊比例0.00%，金额0.00
- **删除按钮**：仅1行数据时提示[至少保留一条分摊信息]；多行时弹框确认后删除
- **第1行数据**的分摊比例和分摊金额不可手动修改，通过联动自动带出
- 修改第2+行分摊比例时，自动计算：第1行 = 100% - Σ（第2+行分摊比例）
- 若Σ（第2+行）> 100%，输入框清空值，提示错误
- 若仅有第1行，第1行默认展示为100%（永远不可编辑）

#### 3.2.9 备注信息分区

- 支持手动输入备注，字数不限
- 展示字符计数提示

#### 3.2.10 单据状态与按钮逻辑

| 操作 | 触发条件 | 单据状态变更 |
| --- | --- | --- |
| 新增 | 点击新增按钮进入页面 | 草稿(code:0) |
| 保存草稿 | 任意时刻点击保存 | 保持草稿(code:0) |
| 提交 | 所有必填项填写完成后点击提交 | 已完成(code:1) |
| 作废 | 已完成状态下点击作废 | 已作废(code:2) |

#### 3.2.11 异常处置

- **必填字段未填写**：提交时前端校验，字段旁红色提示【该项为必填项】
- **行程时间重复**：保存补录行程时校验，弹框提示【存在重复行程时间，请检查】
- **分摊比例超过100%**：实时校验，输入框清空并提示【分摊比例之和不可超过100%】
- **接口调用失败**：弹出错误提示，提供重试机制，日志记录详细错误信息
- **并发保存冲突**：使用乐观锁机制（version字段），冲突时提示【数据已被他人修改，请刷新后重试】

## 5. 关键技术点

### 5.1 并发编程

- **补助日历数据生成**：当补录行程条数较多时，使用CompletableFuture异步并行生成各行程的补助日历数据，提升响应速度
- **分布式锁**：报销单提交时，使用Redis分布式锁（redisson）防止重复提交
- **线程安全**：所有共享状态使用线程安全对象（AtomicLong、ConcurrentHashMap等）

### 5.2 事务控制

#### 5.2.1 数据库局部事务

以下场景需加 `@Transactional` 注解保证原子性：

- **保存报销单**：主表 + 补录行程 + 补助信息 + 补助日历 + 费用分摊 需在同一事务中
- **删除补录行程**：级联删除补助信息和补助日历需在同一事务中
- **提交报销单**：状态变更 + 日志记录需在同一事务中

#### 5.2.2 分布式事务

报销单提交后若需推送BPM审批流，采用最终一致性方案：

- 本地事务保存报销单数据，同时写入待推送MQ消息表
- 通过补偿机制（定时Job扫描）保证MQ消息最终成功发送
- BPM回调失败时，系统自动重试，记录重试日志

### 5.3 Job使用

- **补偿Job**：每5分钟扫描未成功推送BPM的报销单，进行重推（扫描条件：status=待推送 AND create_time < NOW()-5min）
- **避免大数据集**：每次最多扫描100条，分批处理，防止OOM
- **执行周期**：基于XXL-Job配置，支持动态调整

### 5.4 权限控制

- 员工只能查看/编辑自己的报销单（数据隔离：WHERE reimburser_id = current_user_id）
- 财务人员可查看本公司所有报销单
- 管理员可查看所有公司报销单
- A公司员工不能访问B公司的数据，通过company_id字段过滤

### 5.5 Redis使用

Redis键命名规范：

| 用途 | Key格式 | 数据结构 | 过期时间 |
| --- | --- | --- | --- |
| 城市列表缓存 | fk:reim:city:list | String (JSON) | 24小时 |
| 业务类型缓存 | fk:reim:businessType:tree | String (JSON) | 1小时 |
| 员工信息缓存 | fk:reim:employee:{id} | String (JSON) | 30分钟 |
| 分布式锁（提交） | fk:reim:lock:submit:{reimId} | String | 30秒 |

- **避免大Key**：城市列表数据量大时，按省份分片存储
- **防缓存穿透**：不存在的Key使用空值缓存（过期时间5分钟）

### 5.6 敏感信息处理

- 员工工号、银行卡号等敏感字段在日志中脱敏展示
- 接口返回的敏感信息使用@JsonEncrypt注解进行加密处理
- 数据库中不存储明文密码

### 5.7 错误码使用

| 错误码 | 含义 | 处理方式 |
| --- | --- | --- |
| REIM_001 | 报销单不存在 | 返回404，前端跳转列表页 |
| REIM_002 | 报销单状态不允许该操作 | 返回400，前端提示具体原因 |
| REIM_003 | 行程时间重复 | 返回400，高亮冲突行程 |
| REIM_004 | 分摊比例超过100% | 返回400，前端清空输入 |
| REIM_005 | 必填字段缺失 | 返回400，前端标红对应字段 |

### 5.8 异动日志

- 报销单每次状态变更、关键字段修改均需记录异动日志（表：fk_reim_log）
- 日志字段：操作类型、操作人、操作时间、修改前值、修改后值、备注
- 使用AOP切面统一记录，避免侵入业务代码

### 5.9 大数据量问题

- 报销单列表分页查询，禁止全量查询
- 补助日历数据量大时（超过366条），分批查询和处理
- 导出功能使用异步导出，避免接口超时

### 5.10 缓存的使用

- 城市数据、业务类型树形数据为基础静态数据，采用本地缓存+Redis二级缓存
- 缓存更新策略：基础数据变更时主动推送缓存失效消息

### 5.11 MQ的使用

- 报销单提交后，通过MQ异步推送BPM启动审批流
- 审批通过后，通过MQ通知财务系统进行资金结算
- Topic命名：reim.submit.topic、reim.approve.topic

### 5.12 重试机制

- 调用BPM接口失败时，最多重试3次，间隔时间：1s、3s、10s（指数退避）
- 重试超过3次后，记录失败日志并告警，人工介入处理

| 保存补录行程 | 前端 → 接口网关 → 报销服务 → MySQL(fk_reim_trip INSERT/UPDATE + fk_reim_subsidy INSERT + fk_reim_calendar INSERT) → 事务提交 |
| 修改补助日历 | 前端 → 接口网关 → 报销服务 → MySQL(fk_reim_calendar UPDATE) → 联动更新fk_reim_subsidy金额 → 联动更新fk_reim_main合计金额 |
| 提交报销单 | 前端 → 接口网关 → 报销服务 → Redis(分布式锁) → 数据校验 → MySQL(fk_reim_main UPDATE status) → 写MQ消息表 → 释放锁 → MQ异步推送BPM |
| 查询列表 | 前端 → 接口网关 → 报销服务 → MySQL(fk_reim_main SELECT分页) → 组装VO → 返回前端 |
